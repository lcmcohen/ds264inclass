widget-span span9">			<div class="section post-body">				<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>A <strong>301 Moved Permanently</strong> is an <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Status"><code>HTTP response status code</code></a> indicating that the requested resource has been permanently moved to a new URL provided by the <code>Location</code> response header. The <code>3xx</code> category of response codes are used to indicate redirection messages to the client, such that the client will become aware that a redirection to a different resource or URL should take place.</p> <!--more--><p>It can be a challenge to differentiate between all the possible HTTP response codes and determine the exact cause of a message like the <code>301 Moved Permanently</code> code. There are <em>dozens</em> of possible HTTP status codes used to represent the complex relationship between the client, a web application, a web server, and often multiple third-party web services, so determining the cause of a particular status code can be difficult. In this article we'll examine the <code>301 Moved Permanently</code> code by looking at a few troubleshooting tips, along with some potential fixes for common problems that might be causing this issue, so let's get started!</p> <h2>The Problem is Server-Side</h2> <p>All HTTP response status codes that are in the <code>3xx</code> category are considered <code>redirection messages</code>. Such codes indicate to the user agent (i.e. your web browser) that an additional action is required in order to complete the request and access the desired resource. Unlike gateway related <code>5xx</code> response codes, like the <a href="https://airbrake.io/blog/http-errors/502-bad-gateway-error"><code>502 Bad Gateway Error</code></a> we've looked at recently, which may indicate issues <em>either</em> on an upstream server <em>or</em> the client, the <code>301 Moved Permanently</code> code generally indicates an issue on the actual web server hosting your application.</p> <p>That said, the appearance of a <code>301 Moved Permanently</code> is usually not something that requires much user intervention. Most browsers should automatically detect the <code>301 Moved Permanently</code> response code and process the redirection action automatically. The web server hosting the application should typically include a special <code>Location</code> header as part of the response it sends to the client. This <code>Location</code> header indicates the new URL where the requested resource can be found. For example, if a request comes in to access the URL <code>https://airbrake.io</code>, but the web server is configured to forces redirection to a secure version using <code>https</code>, the server response will include the <code>Location: https://airbrake.io</code> header. This tells the browser that it should redirect this request (as well as all future ones) to <code>https://airbrake.io</code> to the secured URL of <code>https://airbrake.io</code>. In most cases, the browser will automatically detect this <code>301 Moved Permanently</code> response code, read the new <code>Location</code> URL, and redirect the request to that new location. It is considered best practice to use a <code>301 Moved Permanently</code> redirection to transition a user agent from HTTP to the secure HTTPS. Thus, if you attempt to go to the insecure URL of <a href="https://airbrake.io">https://airbrake.io</a> right now, you'll automatically be redirected to the HTTPS version of the site (<a href="https://airbrake.io">https://airbrake.io</a>).</p> <p>Since the <code>301 Moved Permanently</code> indicates that something has gone wrong within the <code>server</code> of your application, we can largely disregard the <code>client</code> side of things. If you're trying to diagnose an issue with your own application, you can immediately ignore most client-side code and components, such as HTML, cascading style sheets (CSS), client-side JavaScript, and so forth. This doesn't apply <em>solely</em> to web sites, either. Many smart phone apps that have a modern looking user interface are actually powered by a normal web application behind the scenes; one that is simply hidden from the user. If you're using such an application and a <code>301 Moved Permanently</code> occurs, the issue isn't going to be related to the app installed on your phone or local testing device. Instead, it will be something on the server-side, which is performing most of the logic and processing behind the scenes, outside the purview of the local interface presented to the user.</p> <p>All of that said, if your application is generating <code>301 Moved Permanently</code> codes improperly or unexpectedly, there are a number of steps you can take to diagnose the problem.</p> <h2>Start With a Thorough Application Backup</h2> <p>As with anything, it's better to have played it safe at the start than to screw something up and come to regret it later on down the road. As such, it is <em>critical</em> that you perform a full backup of your application, database, and so forth, before attempting any fixes or changes to the system. Even better, if you have the capability, create a complete copy of the application onto a secondary <code>staging</code> server that isn't "live," or isn't otherwise active and available to the public. This will give you a clean testing ground with which to test all potential fixes to resolve the issue, without threatening the security or sanctity of your live application.</p> <h2>Diagnosing a 301 Moved Permanently Response Code</h2> <p>A <code>301 Moved Permanently</code> response code indicates that the server believes that the requested resource is invalid and that the request should be redirected to a new, "proper" URL. I use the word <code>believes</code> here because it's entirely possible that the server is misconfigured or bugged in some way, which is causing it to provide <code>301 Moved Permanently</code> codes for resources/URLs that are totally valid. Thus, a large part of diagnosing the issue will be going through the process of double-checking what resources/URLs are generating <code>301 Moved Permanently</code> response codes and determining if these codes are appropriate or not.</p> <p>That said, if your application is responding with <code>301 Moved Permanently</code> codes that it <em>should not</em> be issuing, this is an issue that many other visitors may be experiencing as well, dramatically hindering your application's ability to service users. We'll go over some troubleshooting tips and tricks to help you try to resolve this issue. If nothing here works, don't forget that Google is your friend. Don't be afraid to search for specific terms related to your issue, such as the name of your application's CMS or web server software, along with <code>301 Moved Permanently</code>. Chances are you'll find others who have experienced this issue and have come across a solution.</p> <h2>Troubleshooting on the Server-Side</h2> <p>Here are some additional tips to help you troubleshoot what might be causing the <code>301 Moved Permanently</code> to appear on the server-side of things:</p> <ul>  <li><code>Check the Server Configuration Files</code> - Your application is likely running on a server that is using one of the two most popular web server softwares, <code>Apache</code> or <code>nginx</code>. At the time of publication, both of these web servers make up <a href="https://w3techs.com/technologies/overview/web_server/all"><code>over 84%</code> of the world's web server software</a>! Thus, one of the first steps you can take to determine what might be causing these <code>301 Moved Permanently</code> response codes is to check the configuration files for your web server software for unintentional redirect instructions.</li> </ul> <p>To determine which web server your application is using you'll want to look for a key file. If your web server is Apache then look for an <code>.htaccess</code> file within the root directory of your website file system. For example, if your application is on a shared host you'll likely have a username associated with the account on that host. In such a case, the application root directory is likely something like <code>/home/&lt;username&gt;/public_html/</code>, so the <code>.htaccess</code> file would be at <code>/home/&lt;username&gt;/public_html/.htaccess</code>.</p> <p>If you located the <code>.htaccess</code> file then open it in a text editor and look for lines that use <code>RewriteXXX</code> directives, which are part of the <a href="http://httpd.apache.org/docs/current/mod/mod_rewrite.html"><code>mod_rewrite</code></a> module in Apache. Covering exactly how these rules work is well beyond the scope of this article, however, the basic concept is that a <code>RewriteCond</code> directive defines a text-based pattern that will be matched against entered URLs. If a matching URL is requested by a visitor to the site, the <code>RewriteRule</code> directive that follows one or more <code>RewriteCond</code> directives is used to perform the actual redirection of the request to the appropriate URL. Therefore, if you find any strange <code>RewriteCond</code> or <code>RewriteRule</code> directives in the <code>.htaccess</code> file that don't seem to belong, try temporarily commenting them out (using the <code>#</code> character prefix) and restarting your web server to see if this resolves the issue.</p> <p>On the other hand, if your server is running on <code>nginx</code>, you'll need to look for a completely different configuration file. By default this file is named <code>nginx.conf</code> and is located in one of a few common directories: <code>/usr/local/nginx/conf</code>, <code>/etc/nginx</code>, or <code>/usr/local/etc/nginx</code>. Once located, open <code>nginx.conf</code> in a text editor and look for the <code>return</code> or <code>rewrite</code> directives. For example, here is a simple <code>block directive</code> (i.e. a named set of directives) that configures a virtual server by creating a redirection from <code>invalid-domain.com</code> to the proper <code>valid-domain.com</code> URL:</p> <pre><code>server {<br>    listen 80;<br>    listen 443 ssl;<br>    server_name invalid-domain.com;<br>    return 301 $scheme://valid-domain.com$request_uri;<br>}<br></code></pre> <p><code>Rewrite</code> directives in <code>nginx</code> are similar to the <code>RewriteCond</code> and <code>RewriteRule</code> directives found in <code>Apache</code>, as they tend to contain more complex text-based patterns for searching. Either way, look through your <code>nginx.conf</code> file for any abnormal <code>return</code> or <code>rewrite</code> directives and comment them out before restarting the server to see if the issue was resolved.</p> <ul>  <li><code>Check the Logs</code> - Nearly every web application will keep some form of server-side logs. <code>Application logs</code> are typically the history of what the application did, such as which pages were requested, which servers it connected to, which database results it provides, and so forth. <code>Server logs</code> are related to the actual hardware that is running the application, and will often provide details about the health and status of all connected services, or even just the server itself. Google "logs [PLATFORM_NAME]" if you're using a CMS, or "logs [PROGRAMMING_LANGUAGE]" and "logs [OPERATING_SYSTEM]" if you're running a custom application, to get more information on finding the logs in question.</li>  <li><code>Application Code or Script Bugs</code> - If all else fails, it may be that a problem in some custom code within your application is causing the issue. Try to diagnose where the issue may be coming from through manually debugging your application, along with parsing through application and server logs. Ideally, make a copy of the entire application to a local development machine and perform a step-by-step debug process, which will allow you to recreate the exact scenario in which the <code>301 Moved Permanently</code> occurred and view the application code at the moment something goes wrong.</li> </ul> <p>No matter what the cause, the appearance of a <code>301 Moved Permanently</code> within your own web application is a strong indication that you may need an error management tool to help you automatically detect such errors in the future. The best of these tools can even alert you and your team <em>immediately</em> when an error occurs. widget-span span9">			<div class="section post-body">				<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>A <strong>301 Moved Permanently</strong> is an <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Status"><code>HTTP response status code</code></a> indicating that the requested resource has been permanently moved to a new URL provided by the <code>Location</code> response header. The <code>3xx</code> category of response codes are used to indicate redirection messages to the client, such that the client will become aware that a redirection to a different resource or URL should take place.</p> <!--more--><p>It can be a challenge to differentiate between all the possible HTTP response codes and determine the exact cause of a message like the <code>301 Moved Permanently</code> code. There are <em>dozens</em> of possible HTTP status codes used to represent the complex relationship between the client, a web application, a web server, and often multiple third-party web services, so determining the cause of a particular status code can be difficult. In this article we'll examine the <code>301 Moved Permanently</code> code by looking at a few troubleshooting tips, along with some potential fixes for common problems that might be causing this issue, so let's get started!</p> <h2>The Problem is Server-Side</h2> <p>All HTTP response status codes that are in the <code>3xx</code> category are considered <code>redirection messages</code>. Such codes indicate to the user agent (i.e. your web browser) that an additional action is required in order to complete the request and access the desired resource. Unlike gateway related <code>5xx</code> response codes, like the <a href="https://airbrake.io/blog/http-errors/502-bad-gateway-error"><code>502 Bad Gateway Error</code></a> we've looked at recently, which may indicate issues <em>either</em> on an upstream server <em>or</em> the client, the <code>301 Moved Permanently</code> code generally indicates an issue on the actual web server hosting your application.</p> <p>That said, the appearance of a <code>301 Moved Permanently</code> is usually not something that requires much user intervention. Most browsers should automatically detect the <code>301 Moved Permanently</code> response code and process the redirection action automatically. The web server hosting the application should typically include a special <code>Location</code> header as part of the response it sends to the client. This <code>Location</code> header indicates the new URL where the requested resource can be found. For example, if a request comes in to access the URL <code>https://airbrake.io</code>, but the web server is configured to forces redirection to a secure version using <code>https</code>, the server response will include the <code>Location: https://airbrake.io</code> header. This tells the browser that it should redirect this request (as well as all future ones) to <code>https://airbrake.io</code> to the secured URL of <code>https://airbrake.io</code>. In most cases, the browser will automatically detect this <code>301 Moved Permanently</code> response code, read the new <code>Location</code> URL, and redirect the request to that new location. It is considered best practice to use a <code>301 Moved Permanently</code> redirection to transition a user agent from HTTP to the secure HTTPS. Thus, if you attempt to go to the insecure URL of <a href="https://airbrake.io">https://airbrake.io</a> right now, you'll automatically be redirected to the HTTPS version of the site (<a href="https://airbrake.io">https://airbrake.io</a>).</p> <p>Since the <code>301 Moved Permanently</code> indicates that something has gone wrong within the <code>server</code> of your application, we can largely disregard the <code>client</code> side of things. If you're trying to diagnose an issue with your own application, you can immediately ignore most client-side code and components, such as HTML, cascading style sheets (CSS), client-side JavaScript, and so forth. This doesn't apply <em>solely</em> to web sites, either. Many smart phone apps that have a modern looking user interface are actually powered by a normal web application behind the scenes; one that is simply hidden from the user. If you're using such an application and a <code>301 Moved Permanently</code> occurs, the issue isn't going to be related to the app installed on your phone or local testing device. Instead, it will be something on the server-side, which is performing most of the logic and processing behind the scenes, outside the purview of the local interface presented to the user.</p> <p>All of that said, if your application is generating <code>301 Moved Permanently</code> codes improperly or unexpectedly, there are a number of steps you can take to diagnose the problem.</p> <h2>Start With a Thorough Application Backup</h2> <p>As with anything, it's better to have played it safe at the start than to screw something up and come to regret it later on down the road. As such, it is <em>critical</em> that you perform a full backup of your application, database, and so forth, before attempting any fixes or changes to the system. Even better, if you have the capability, create a complete copy of the application onto a secondary <code>staging</code> server that isn't "live," or isn't otherwise active and available to the public. This will give you a clean testing ground with which to test all potential fixes to resolve the issue, without threatening the security or sanctity of your live application.</p> <h2>Diagnosing a 301 Moved Permanently Response Code</h2> <p>A <code>301 Moved Permanently</code> response code indicates that the server believes that the requested resource is invalid and that the request should be redirected to a new, "proper" URL. I use the word <code>believes</code> here because it's entirely possible that the server is misconfigured or bugged in some way, which is causing it to provide <code>301 Moved Permanently</code> codes for resources/URLs that are totally valid. Thus, a large part of diagnosing the issue will be going through the process of double-checking what resources/URLs are generating <code>301 Moved Permanently</code> response codes and determining if these codes are appropriate or not.</p> <p>That said, if your application is responding with <code>301 Moved Permanently</code> codes that it <em>should not</em> be issuing, this is an issue that many other visitors may be experiencing as well, dramatically hindering your application's ability to service users. We'll go over some troubleshooting tips and tricks to help you try to resolve this issue. If nothing here works, don't forget that Google is your friend. Don't be afraid to search for specific terms related to your issue, such as the name of your application's CMS or web server software, along with <code>301 Moved Permanently</code>. Chances are you'll find others who have experienced this issue and have come across a solution.</p> <h2>Troubleshooting on the Server-Side</h2> <p>Here are some additional tips to help you troubleshoot what might be causing the <code>301 Moved Permanently</code> to appear on the server-side of things:</p> <ul>  <li><code>Check the Server Configuration Files</code> - Your application is likely running on a server that is using one of the two most popular web server softwares, <code>Apache</code> or <code>nginx</code>. At the time of publication, both of these web servers make up <a href="https://w3techs.com/technologies/overview/web_server/all"><code>over 84%</code> of the world's web server software</a>! Thus, one of the first steps you can take to determine what might be causing these <code>301 Moved Permanently</code> response codes is to check the configuration files for your web server software for unintentional redirect instructions.</li> </ul> <p>To determine which web server your application is using you'll want to look for a key file. If your web server is Apache then look for an <code>.htaccess</code> file within the root directory of your website file system. For example, if your application is on a shared host you'll likely have a username associated with the account on that host. In such a case, the application root directory is likely something like <code>/home/&lt;username&gt;/public_html/</code>, so the <code>.htaccess</code> file would be at <code>/home/&lt;username&gt;/public_html/.htaccess</code>.</p> <p>If you located the <code>.htaccess</code> file then open it in a text editor and look for lines that use <code>RewriteXXX</code> directives, which are part of the <a href="http://httpd.apache.org/docs/current/mod/mod_rewrite.html"><code>mod_rewrite</code></a> module in Apache. Covering exactly how these rules work is well beyond the scope of this article, however, the basic concept is that a <code>RewriteCond</code> directive defines a text-based pattern that will be matched against entered URLs. If a matching URL is requested by a visitor to the site, the <code>RewriteRule</code> directive that follows one or more <code>RewriteCond</code> directives is used to perform the actual redirection of the request to the appropriate URL. Therefore, if you find any strange <code>RewriteCond</code> or <code>RewriteRule</code> directives in the <code>.htaccess</code> file that don't seem to belong, try temporarily commenting them out (using the <code>#</code> character prefix) and restarting your web server to see if this resolves the issue.</p> <p>On the other hand, if your server is running on <code>nginx</code>, you'll need to look for a completely different configuration file. By default this file is named <code>nginx.conf</code> and is located in one of a few common directories: <code>/usr/local/nginx/conf</code>, <code>/etc/nginx</code>, or <code>/usr/local/etc/nginx</code>. Once located, open <code>nginx.conf</code> in a text editor and look for the <code>return</code> or <code>rewrite</code> directives. For example, here is a simple <code>block directive</code> (i.e. a named set of directives) that configures a virtual server by creating a redirection from <code>invalid-domain.com</code> to the proper <code>valid-domain.com</code> URL:</p> <pre><code>server {<br>    listen 80;<br>    listen 443 ssl;<br>    server_name invalid-domain.com;<br>    return 301 $scheme://valid-domain.com$request_uri;<br>}<br></code></pre> <p><code>Rewrite</code> directives in <code>nginx</code> are similar to the <code>RewriteCond</code> and <code>RewriteRule</code> directives found in <code>Apache</code>, as they tend to contain more complex text-based patterns for searching. Either way, look through your <code>nginx.conf</code> file for any abnormal <code>return</code> or <code>rewrite</code> directives and comment them out before restarting the server to see if the issue was resolved.</p> <ul>  <li><code>Check the Logs</code> - Nearly every web application will keep some form of server-side logs. <code>Application logs</code> are typically the history of what the application did, such as which pages were requested, which servers it connected to, which database results it provides, and so forth. <code>Server logs</code> are related to the actual hardware that is running the application, and will often provide details about the health and status of all connected services, or even just the server itself. Google "logs [PLATFORM_NAME]" if you're using a CMS, or "logs [PROGRAMMING_LANGUAGE]" and "logs [OPERATING_SYSTEM]" if you're running a custom application, to get more information on finding the logs in question.</li>  <li><code>Application Code or Script Bugs</code> - If all else fails, it may be that a problem in some custom code within your application is causing the issue. Try to diagnose where the issue may be coming from through manually debugging your application, along with parsing through application and server logs. Ideally, make a copy of the entire application to a local development machine and perform a step-by-step debug process, which will allow you to recreate the exact scenario in which the <code>301 Moved Permanently</code> occurred and view the application code at the moment something goes wrong.</li> </ul> <p>No matter what the cause, the appearance of a <code>301 Moved Permanently</code> within your own web application is a strong indication that you may need an error management tool to help you automatically detect such errors in the future. data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>A <strong>301 Moved Permanently</strong> is an <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Status"><code>HTTP response status code</code></a> indicating that the requested resource has been permanently moved to a new URL provided by the <code>Location</code> response header. The <code>3xx</code> category of response codes are used to indicate redirection messages to the client, such that the client will become aware that a redirection to a different resource or URL should take place.</p> <!--more--><p>It can be a challenge to differentiate between all the possible HTTP response codes and determine the exact cause of a message like the <code>301 Moved Permanently</code> code. There are <em>dozens</em> of possible HTTP status codes used to represent the complex relationship between the client, a web application, a web server, and often multiple third-party web services, so determining the cause of a particular status code can be difficult. In this article we'll examine the <code>301 Moved Permanently</code> code by looking at a few troubleshooting tips, along with some potential fixes for common problems that might be causing this issue, so let's get started!</p> <h2>The Problem is Server-Side</h2> <p>All HTTP response status codes that are in the <code>3xx</code> category are considered <code>redirection messages</code>. Such codes indicate to the user agent (i.e. your web browser) that an additional action is required in order to complete the request and access the desired resource. Unlike gateway related <code>5xx</code> response codes, like the <a href="https://airbrake.io/blog/http-errors/502-bad-gateway-error"><code>502 Bad Gateway Error</code></a> we've looked at recently, which may indicate issues <em>either</em> on an upstream server <em>or</em> the client, the <code>301 Moved Permanently</code> code generally indicates an issue on the actual web server hosting your application.</p> <p>That said, the appearance of a <code>301 Moved Permanently</code> is usually not something that requires much user intervention. Most browsers should automatically detect the <code>301 Moved Permanently</code> response code and process the redirection action automatically. The web server hosting the application should typically include a special <code>Location</code> header as part of the response it sends to the client. This <code>Location</code> header indicates the new URL where the requested resource can be found. For example, if a request comes in to access the URL <code>https://airbrake.io</code>, but the web server is configured to forces redirection to a secure version using <code>https</code>, the server response will include the <code>Location: https://airbrake.io</code> header. This tells the browser that it should redirect this request (as well as all future ones) to <code>https://airbrake.io</code> to the secured URL of <code>https://airbrake.io</code>. In most cases, the browser will automatically detect this <code>301 Moved Permanently</code> response code, read the new <code>Location</code> URL, and redirect the request to that new location. It is considered best practice to use a <code>301 Moved Permanently</code> redirection to transition a user agent from HTTP to the secure HTTPS. Thus, if you attempt to go to the insecure URL of <a href="https://airbrake.io">https://airbrake.io</a> right now, you'll automatically be redirected to the HTTPS version of the site (<a href="https://airbrake.io">https://airbrake.io</a>).</p> <p>Since the <code>301 Moved Permanently</code> indicates that something has gone wrong within the <code>server</code> of your application, we can largely disregard the <code>client</code> side of things. If you're trying to diagnose an issue with your own application, you can immediately ignore most client-side code and components, such as HTML, cascading style sheets (CSS), client-side JavaScript, and so forth. This doesn't apply <em>solely</em> to web sites, either. Many smart phone apps that have a modern looking user interface are actually powered by a normal web application behind the scenes; one that is simply hidden from the user. If you're using such an application and a <code>301 Moved Permanently</code> occurs, the issue isn't going to be related to the app installed on your phone or local testing device. Instead, it will be something on the server-side, which is performing most of the logic and processing behind the scenes, outside the purview of the local interface presented to the user.</p> <p>All of that said, if your application is generating <code>301 Moved Permanently</code> codes improperly or unexpectedly, there are a number of steps you can take to diagnose the problem.</p> <h2>Start With a Thorough Application Backup</h2> <p>As with anything, it's better to have played it safe at the start than to screw something up and come to regret it later on down the road. As such, it is <em>critical</em> that you perform a full backup of your application, database, and so forth, before attempting any fixes or changes to the system. Even better, if you have the capability, create a complete copy of the application onto a secondary <code>staging</code> server that isn't "live," or isn't otherwise active and available to the public. This will give you a clean testing ground with which to test all potential fixes to resolve the issue, without threatening the security or sanctity of your live application.</p> <h2>Diagnosing a 301 Moved Permanently Response Code</h2> <p>A <code>301 Moved Permanently</code> response code indicates that the server believes that the requested resource is invalid and that the request should be redirected to a new, "proper" URL. I use the word <code>believes</code> here because it's entirely possible that the server is misconfigured or bugged in some way, which is causing it to provide <code>301 Moved Permanently</code> codes for resources/URLs that are totally valid. Thus, a large part of diagnosing the issue will be going through the process of double-checking what resources/URLs are generating <code>301 Moved Permanently</code> response codes and determining if these codes are appropriate or not.</p> <p>That said, if your application is responding with <code>301 Moved Permanently</code> codes that it <em>should not</em> be issuing, this is an issue that many other visitors may be experiencing as well, dramatically hindering your application's ability to service users. We'll go over some troubleshooting tips and tricks to help you try to resolve this issue. If nothing here works, don't forget that Google is your friend. Don't be afraid to search for specific terms related to your issue, such as the name of your application's CMS or web server software, along with <code>301 Moved Permanently</code>. Chances are you'll find others who have experienced this issue and have come across a solution.</p> <h2>Troubleshooting on the Server-Side</h2> <p>Here are some additional tips to help you troubleshoot what might be causing the <code>301 Moved Permanently</code> to appear on the server-side of things:</p> <ul>  <li><code>Check the Server Configuration Files</code> - Your application is likely running on a server that is using one of the two most popular web server softwares, <code>Apache</code> or <code>nginx</code>. At the time of publication, both of these web servers make up <a href="https://w3techs.com/technologies/overview/web_server/all"><code>over 84%</code> of the world's web server software</a>! Thus, one of the first steps you can take to determine what might be causing these <code>301 Moved Permanently</code> response codes is to check the configuration files for your web server software for unintentional redirect instructions.</li> </ul> <p>To determine which web server your application is using you'll want to look for a key file. If your web server is Apache then look for an <code>.htaccess</code> file within the root directory of your website file system. For example, if your application is on a shared host you'll likely have a username associated with the account on that host. In such a case, the application root directory is likely something like <code>/home/&lt;username&gt;/public_html/</code>, so the <code>.htaccess</code> file would be at <code>/home/&lt;username&gt;/public_html/.htaccess</code>.</p> <p>If you located the <code>.htaccess</code> file then open it in a text editor and look for lines that use <code>RewriteXXX</code> directives, which are part of the <a href="http://httpd.apache.org/docs/current/mod/mod_rewrite.html"><code>mod_rewrite</code></a> module in Apache. Covering exactly how these rules work is well beyond the scope of this article, however, the basic concept is that a <code>RewriteCond</code> directive defines a text-based pattern that will be matched against entered URLs. If a matching URL is requested by a visitor to the site, the <code>RewriteRule</code> directive that follows one or more <code>RewriteCond</code> directives is used to perform the actual redirection of the request to the appropriate URL. Therefore, if you find any strange <code>RewriteCond</code> or <code>RewriteRule</code> directives in the <code>.htaccess</code> file that don't seem to belong, try temporarily commenting them out (using the <code>#</code> character prefix) and restarting your web server to see if this resolves the issue.</p> <p>On the other hand, if your server is running on <code>nginx</code>, you'll need to look for a completely different configuration file. By default this file is named <code>nginx.conf</code> and is located in one of a few common directories: <code>/usr/local/nginx/conf</code>, <code>/etc/nginx</code>, or <code>/usr/local/etc/nginx</code>. Once located, open <code>nginx.conf</code> in a text editor and look for the <code>return</code> or <code>rewrite</code> directives. For example, here is a simple <code>block directive</code> (i.e. a named set of directives) that configures a virtual server by creating a redirection from <code>invalid-domain.com</code> to the proper <code>valid-domain.com</code> URL:</p> <pre><code>server {<br>    listen 80;<br>    listen 443 ssl;<br>    server_name invalid-domain.com;<br>    return 301 $scheme://valid-domain.com$request_uri;<br>}<br></code></pre> <p><code>Rewrite</code> directives in <code>nginx</code> are similar to the <code>RewriteCond</code> and <code>RewriteRule</code> directives found in <code>Apache</code>, as they tend to contain more complex text-based patterns for searching. Either way, look through your <code>nginx.conf</code> file for any abnormal <code>return</code> or <code>rewrite</code> directives and comment them out before restarting the server to see if the issue was resolved.</p> <ul>  <li><code>Check the Logs</code> - Nearly every web application will keep some form of server-side logs. <code>Application logs</code> are typically the history of what the application did, such as which pages were requested, which servers it connected to, which database results it provides, and so forth. <code>Server logs</code> are related to the actual hardware that is running the application, and will often provide details about the health and status of all connected services, or even just the server itself. Google "logs [PLATFORM_NAME]" if you're using a CMS, or "logs [PROGRAMMING_LANGUAGE]" and "logs [OPERATING_SYSTEM]" if you're running a custom application, to get more information on finding the logs in question.</li>  <li><code>Application Code or Script Bugs</code> - If all else fails, it may be that a problem in some custom code within your application is causing the issue. Try to diagnose where the issue may be coming from through manually debugging your application, along with parsing through application and server logs. Ideally, make a copy of the entire application to a local development machine and perform a step-by-step debug process, which will allow you to recreate the exact scenario in which the <code>301 Moved Permanently</code> occurred and view the application code at the moment something goes wrong.</li> </ul> <p>No matter what the cause, the appearance of a <code>301 Moved Permanently</code> within your own web application is a strong indication that you may need an error management tool to help you automatically detect such errors in the future. data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>A <strong>301 Moved Permanently</strong> is an <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Status"><code>HTTP response status code</code></a> indicating that the requested resource has been permanently moved to a new URL provided by the <code>Location</code> response header. The <code>3xx</code> category of response codes are used to indicate redirection messages to the client, such that the client will become aware that a redirection to a different resource or URL should take place.</p> <!--more--><p>It can be a challenge to differentiate between all the possible HTTP response codes and determine the exact cause of a message like the <code>301 Moved Permanently</code> code. There are <em>dozens</em> of possible HTTP status codes used to represent the complex relationship between the client, a web application, a web server, and often multiple third-party web services, so determining the cause of a particular status code can be difficult. In this article we'll examine the <code>301 Moved Permanently</code> code by looking at a few troubleshooting tips, along with some potential fixes for common problems that might be causing this issue, so let's get started!</p> <h2>The Problem is Server-Side</h2> <p>All HTTP response status codes that are in the <code>3xx</code> category are considered <code>redirection messages</code>. Such codes indicate to the user agent (i.e. your web browser) that an additional action is required in order to complete the request and access the desired resource. Unlike gateway related <code>5xx</code> response codes, like the <a href="https://airbrake.io/blog/http-errors/502-bad-gateway-error"><code>502 Bad Gateway Error</code></a> we've looked at recently, which may indicate issues <em>either</em> on an upstream server <em>or</em> the client, the <code>301 Moved Permanently</code> code generally indicates an issue on the actual web server hosting your application.</p> <p>That said, the appearance of a <code>301 Moved Permanently</code> is usually not something that requires much user intervention. Most browsers should automatically detect the <code>301 Moved Permanently</code> response code and process the redirection action automatically. The web server hosting the application should typically include a special <code>Location</code> header as part of the response it sends to the client. This <code>Location</code> header indicates the new URL where the requested resource can be found. For example, if a request comes in to access the URL <code>https://airbrake.io</code>, but the web server is configured to forces redirection to a secure version using <code>https</code>, the server response will include the <code>Location: https://airbrake.io</code> header. This tells the browser that it should redirect this request (as well as all future ones) to <code>https://airbrake.io</code> to the secured URL of <code>https://airbrake.io</code>. In most cases, the browser will automatically detect this <code>301 Moved Permanently</code> response code, read the new <code>Location</code> URL, and redirect the request to that new location. It is considered best practice to use a <code>301 Moved Permanently</code> redirection to transition a user agent from HTTP to the secure HTTPS. Thus, if you attempt to go to the insecure URL of <a href="https://airbrake.io">https://airbrake.io</a> right now, you'll automatically be redirected to the HTTPS version of the site (<a href="https://airbrake.io">https://airbrake.io</a>).</p> <p>Since the <code>301 Moved Permanently</code> indicates that something has gone wrong within the <code>server</code> of your application, we can largely disregard the <code>client</code> side of things. If you're trying to diagnose an issue with your own application, you can immediately ignore most client-side code and components, such as HTML, cascading style sheets (CSS), client-side JavaScript, and so forth. This doesn't apply <em>solely</em> to web sites, either. Many smart phone apps that have a modern looking user interface are actually powered by a normal web application behind the scenes; one that is simply hidden from the user. If you're using such an application and a <code>301 Moved Permanently</code> occurs, the issue isn't going to be related to the app installed on your phone or local testing device. Instead, it will be something on the server-side, which is performing most of the logic and processing behind the scenes, outside the purview of the local interface presented to the user.</p> <p>All of that said, if your application is generating <code>301 Moved Permanently</code> codes improperly or unexpectedly, there are a number of steps you can take to diagnose the problem.</p> <h2>Start With a Thorough Application Backup</h2> <p>As with anything, it's better to have played it safe at the start than to screw something up and come to regret it later on down the road. As such, it is <em>critical</em> that you perform a full backup of your application, database, and so forth, before attempting any fixes or changes to the system. Even better, if you have the capability, create a complete copy of the application onto a secondary <code>staging</code> server that isn't "live," or isn't otherwise active and available to the public. This will give you a clean testing ground with which to test all potential fixes to resolve the issue, without threatening the security or sanctity of your live application.</p> <h2>Diagnosing a 301 Moved Permanently Response Code</h2> <p>A <code>301 Moved Permanently</code> response code indicates that the server believes that the requested resource is invalid and that the request should be redirected to a new, "proper" URL. I use the word <code>believes</code> here because it's entirely possible that the server is misconfigured or bugged in some way, which is causing it to provide <code>301 Moved Permanently</code> codes for resources/URLs that are totally valid. Thus, a large part of diagnosing the issue will be going through the process of double-checking what resources/URLs are generating <code>301 Moved Permanently</code> response codes and determining if these codes are appropriate or not.</p> <p>That said, if your application is responding with <code>301 Moved Permanently</code> codes that it <em>should not</em> be issuing, this is an issue that many other visitors may be experiencing as well, dramatically hindering your application's ability to service users. We'll go over some troubleshooting tips and tricks to help you try to resolve this issue. If nothing here works, don't forget that Google is your friend. Don't be afraid to search for specific terms related to your issue, such as the name of your application's CMS or web server software, along with <code>301 Moved Permanently</code>. Chances are you'll find others who have experienced this issue and have come across a solution.</p> <h2>Troubleshooting on the Server-Side</h2> <p>Here are some additional tips to help you troubleshoot what might be causing the <code>301 Moved Permanently</code> to appear on the server-side of things:</p> <ul>  <li><code>Check the Server Configuration Files</code> - Your application is likely running on a server that is using one of the two most popular web server softwares, <code>Apache</code> or <code>nginx</code>. At the time of publication, both of these web servers make up <a href="https://w3techs.com/technologies/overview/web_server/all"><code>over 84%</code> of the world's web server software</a>! Thus, one of the first steps you can take to determine what might be causing these <code>301 Moved Permanently</code> response codes is to check the configuration files for your web server software for unintentional redirect instructions.</li> </ul> <p>To determine which web server your application is using you'll want to look for a key file. If your web server is Apache then look for an <code>.htaccess</code> file within the root directory of your website file system. For example, if your application is on a shared host you'll likely have a username associated with the account on that host. In such a case, the application root directory is likely something like <code>/home/&lt;username&gt;/public_html/</code>, so the <code>.htaccess</code> file would be at <code>/home/&lt;username&gt;/public_html/.htaccess</code>.</p> <p>If you located the <code>.htaccess</code> file then open it in a text editor and look for lines that use <code>RewriteXXX</code> directives, which are part of the <a href="http://httpd.apache.org/docs/current/mod/mod_rewrite.html"><code>mod_rewrite</code></a> module in Apache. Covering exactly how these rules work is well beyond the scope of this article, however, the basic concept is that a <code>RewriteCond</code> directive defines a text-based pattern that will be matched against entered URLs. If a matching URL is requested by a visitor to the site, the <code>RewriteRule</code> directive that follows one or more <code>RewriteCond</code> directives is used to perform the actual redirection of the request to the appropriate URL. Therefore, if you find any strange <code>RewriteCond</code> or <code>RewriteRule</code> directives in the <code>.htaccess</code> file that don't seem to belong, try temporarily commenting them out (using the <code>#</code> character prefix) and restarting your web server to see if this resolves the issue.</p> <p>On the other hand, if your server is running on <code>nginx</code>, you'll need to look for a completely different configuration file. By default this file is named <code>nginx.conf</code> and is located in one of a few common directories: <code>/usr/local/nginx/conf</code>, <code>/etc/nginx</code>, or <code>/usr/local/etc/nginx</code>. Once located, open <code>nginx.conf</code> in a text editor and look for the <code>return</code> or <code>rewrite</code> directives. For example, here is a simple <code>block directive</code> (i.e. a named set of directives) that configures a virtual server by creating a redirection from <code>invalid-domain.com</code> to the proper <code>valid-domain.com</code> URL:</p> <pre><code>server {<br>    listen 80;<br>    listen 443 ssl;<br>    server_name invalid-domain.com;<br>    return 301 $scheme://valid-domain.com$request_uri;<br>}<br></code></pre> <p><code>Rewrite</code> directives in <code>nginx</code> are similar to the <code>RewriteCond</code> and <code>RewriteRule</code> directives found in <code>Apache</code>, as they tend to contain more complex text-based patterns for searching. Either way, look through your <code>nginx.conf</code> file for any abnormal <code>return</code> or <code>rewrite</code> directives and comment them out before restarting the server to see if the issue was resolved.</p> <ul>  <li><code>Check the Logs</code> - Nearly every web application will keep some form of server-side logs. <code>Application logs</code> are typically the history of what the application did, such as which pages were requested, which servers it connected to, which database results it provides, and so forth. <code>Server logs</code> are related to the actual hardware that is running the application, and will often provide details about the health and status of all connected services, or even just the server itself. Google "logs [PLATFORM_NAME]" if you're using a CMS, or "logs [PROGRAMMING_LANGUAGE]" and "logs [OPERATING_SYSTEM]" if you're running a custom application, to get more information on finding the logs in question.</li>  <li><code>Application Code or Script Bugs</code> - If all else fails, it may be that a problem in some custom code within your application is causing the issue. Try to diagnose where the issue may be coming from through manually debugging your application, along with parsing through application and server logs. Ideally, make a copy of the entire application to a local development machine and perform a step-by-step debug process, which will allow you to recreate the exact scenario in which the <code>301 Moved Permanently</code> occurred and view the application code at the moment something goes wrong.</li> </ul> <p>No matter what the cause, the appearance of a <code>301 Moved Permanently</code> within your own web application is a strong indication that you may need an error management tool to help you automatically detect such errors in the future. data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>A <strong>301 Moved Permanently</strong> is an <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Status"><code>HTTP response status code</code></a> indicating that the requested resource has been permanently moved to a new URL provided by the <code>Location</code> response header. The <code>3xx</code> category of response codes are used to indicate redirection messages to the client, such that the client will become aware that a redirection to a different resource or URL should take place.</p> <!--more--><p>It can be a challenge to differentiate between all the possible HTTP response codes and determine the exact cause of a message like the <code>301 Moved Permanently</code> code. There are <em>dozens</em> of possible HTTP status codes used to represent the complex relationship between the client, a web application, a web server, and often multiple third-party web services, so determining the cause of a particular status code can be difficult. In this article we'll examine the <code>301 Moved Permanently</code> code by looking at a few troubleshooting tips, along with some potential fixes for common problems that might be causing this issue, so let's get started!</p> <h2>The Problem is Server-Side</h2> <p>All HTTP response status codes that are in the <code>3xx</code> category are considered <code>redirection messages</code>. Such codes indicate to the user agent (i.e. your web browser) that an additional action is required in order to complete the request and access the desired resource. Unlike gateway related <code>5xx</code> response codes, like the <a href="https://airbrake.io/blog/http-errors/502-bad-gateway-error"><code>502 Bad Gateway Error</code></a> we've looked at recently, which may indicate issues <em>either</em> on an upstream server <em>or</em> the client, the <code>301 Moved Permanently</code> code generally indicates an issue on the actual web server hosting your application.</p> <p>That said, the appearance of a <code>301 Moved Permanently</code> is usually not something that requires much user intervention. Most browsers should automatically detect the <code>301 Moved Permanently</code> response code and process the redirection action automatically. The web server hosting the application should typically include a special <code>Location</code> header as part of the response it sends to the client. This <code>Location</code> header indicates the new URL where the requested resource can be found. For example, if a request comes in to access the URL <code>https://airbrake.io</code>, but the web server is configured to forces redirection to a secure version using <code>https</code>, the server response will include the <code>Location: https://airbrake.io</code> header. This tells the browser that it should redirect this request (as well as all future ones) to <code>https://airbrake.io</code> to the secured URL of <code>https://airbrake.io</code>. In most cases, the browser will automatically detect this <code>301 Moved Permanently</code> response code, read the new <code>Location</code> URL, and redirect the request to that new location. It is considered best practice to use a <code>301 Moved Permanently</code> redirection to transition a user agent from HTTP to the secure HTTPS. Thus, if you attempt to go to the insecure URL of <a href="https://airbrake.io">https://airbrake.io</a> right now, you'll automatically be redirected to the HTTPS version of the site (<a href="https://airbrake.io">https://airbrake.io</a>).</p> <p>Since the <code>301 Moved Permanently</code> indicates that something has gone wrong within the <code>server</code> of your application, we can largely disregard the <code>client</code> side of things. If you're trying to diagnose an issue with your own application, you can immediately ignore most client-side code and components, such as HTML, cascading style sheets (CSS), client-side JavaScript, and so forth. This doesn't apply <em>solely</em> to web sites, either. Many smart phone apps that have a modern looking user interface are actually powered by a normal web application behind the scenes; one that is simply hidden from the user. If you're using such an application and a <code>301 Moved Permanently</code> occurs, the issue isn't going to be related to the app installed on your phone or local testing device. Instead, it will be something on the server-side, which is performing most of the logic and processing behind the scenes, outside the purview of the local interface presented to the user.</p> <p>All of that said, if your application is generating <code>301 Moved Permanently</code> codes improperly or unexpectedly, there are a number of steps you can take to diagnose the problem.</p> <h2>Start With a Thorough Application Backup</h2> <p>As with anything, it's better to have played it safe at the start than to screw something up and come to regret it later on down the road. As such, it is <em>critical</em> that you perform a full backup of your application, database, and so forth, before attempting any fixes or changes to the system. Even better, if you have the capability, create a complete copy of the application onto a secondary <code>staging</code> server that isn't "live," or isn't otherwise active and available to the public. This will give you a clean testing ground with which to test all potential fixes to resolve the issue, without threatening the security or sanctity of your live application.</p> <h2>Diagnosing a 301 Moved Permanently Response Code</h2> <p>A <code>301 Moved Permanently</code> response code indicates that the server believes that the requested resource is invalid and that the request should be redirected to a new, "proper" URL. I use the word <code>believes</code> here because it's entirely possible that the server is misconfigured or bugged in some way, which is causing it to provide <code>301 Moved Permanently</code> codes for resources/URLs that are totally valid. Thus, a large part of diagnosing the issue will be going through the process of double-checking what resources/URLs are generating <code>301 Moved Permanently</code> response codes and determining if these codes are appropriate or not.</p> <p>That said, if your application is responding with <code>301 Moved Permanently</code> codes that it <em>should not</em> be issuing, this is an issue that many other visitors may be experiencing as well, dramatically hindering your application's ability to service users. We'll go over some troubleshooting tips and tricks to help you try to resolve this issue. If nothing here works, don't forget that Google is your friend. Don't be afraid to search for specific terms related to your issue, such as the name of your application's CMS or web server software, along with <code>301 Moved Permanently</code>. Chances are you'll find others who have experienced this issue and have come across a solution.</p> <h2>Troubleshooting on the Server-Side</h2> <p>Here are some additional tips to help you troubleshoot what might be causing the <code>301 Moved Permanently</code> to appear on the server-side of things:</p> <ul>  <li><code>Check the Server Configuration Files</code> - Your application is likely running on a server that is using one of the two most popular web server softwares, <code>Apache</code> or <code>nginx</code>. At the time of publication, both of these web servers make up <a href="https://w3techs.com/technologies/overview/web_server/all"><code>over 84%</code> of the world's web server software</a>! Thus, one of the first steps you can take to determine what might be causing these <code>301 Moved Permanently</code> response codes is to check the configuration files for your web server software for unintentional redirect instructions.</li> </ul> <p>To determine which web server your application is using you'll want to look for a key file. If your web server is Apache then look for an <code>.htaccess</code> file within the root directory of your website file system. For example, if your application is on a shared host you'll likely have a username associated with the account on that host. In such a case, the application root directory is likely something like <code>/home/&lt;username&gt;/public_html/</code>, so the <code>.htaccess</code> file would be at <code>/home/&lt;username&gt;/public_html/.htaccess</code>.</p> <p>If you located the <code>.htaccess</code> file then open it in a text editor and look for lines that use <code>RewriteXXX</code> directives, which are part of the <a href="http://httpd.apache.org/docs/current/mod/mod_rewrite.html"><code>mod_rewrite</code></a> module in Apache. Covering exactly how these rules work is well beyond the scope of this article, however, the basic concept is that a <code>RewriteCond</code> directive defines a text-based pattern that will be matched against entered URLs. If a matching URL is requested by a visitor to the site, the <code>RewriteRule</code> directive that follows one or more <code>RewriteCond</code> directives is used to perform the actual redirection of the request to the appropriate URL. Therefore, if you find any strange <code>RewriteCond</code> or <code>RewriteRule</code> directives in the <code>.htaccess</code> file that don't seem to belong, try temporarily commenting them out (using the <code>#</code> character prefix) and restarting your web server to see if this resolves the issue.</p> <p>On the other hand, if your server is running on <code>nginx</code>, you'll need to look for a completely different configuration file. By default this file is named <code>nginx.conf</code> and is located in one of a few common directories: <code>/usr/local/nginx/conf</code>, <code>/etc/nginx</code>, or <code>/usr/local/etc/nginx</code>. Once located, open <code>nginx.conf</code> in a text editor and look for the <code>return</code> or <code>rewrite</code> directives. For example, here is a simple <code>block directive</code> (i.e. a named set of directives) that configures a virtual server by creating a redirection from <code>invalid-domain.com</code> to the proper <code>valid-domain.com</code> URL:</p> <pre><code>server {<br>    listen 80;<br>    listen 443 ssl;<br>    server_name invalid-domain.com;<br>    return 301 $scheme://valid-domain.com$request_uri;<br>}<br></code></pre> <p><code>Rewrite</code> directives in <code>nginx</code> are similar to the <code>RewriteCond</code> and <code>RewriteRule</code> directives found in <code>Apache</code>, as they tend to contain more complex text-based patterns for searching. Either way, look through your <code>nginx.conf</code> file for any abnormal <code>return</code> or <code>rewrite</code> directives and comment them out before restarting the server to see if the issue was resolved.</p> <ul>  <li><code>Check the Logs</code> - Nearly every web application will keep some form of server-side logs. <code>Application logs</code> are typically the history of what the application did, such as which pages were requested, which servers it connected to, which database results it provides, and so forth. <code>Server logs</code> are related to the actual hardware that is running the application, and will often provide details about the health and status of all connected services, or even just the server itself. Google "logs [PLATFORM_NAME]" if you're using a CMS, or "logs [PROGRAMMING_LANGUAGE]" and "logs [OPERATING_SYSTEM]" if you're running a custom application, to get more information on finding the logs in question.</li>  <li><code>Application Code or Script Bugs</code> - If all else fails, it may be that a problem in some custom code within your application is causing the issue. Try to diagnose where the issue may be coming from through manually debugging your application, along with parsing through application and server logs. Ideally, make a copy of the entire application to a local development machine and perform a step-by-step debug process, which will allow you to recreate the exact scenario in which the <code>301 Moved Permanently</code> occurred and view the application code at the moment something goes wrong.</li> </ul> <p>No matter what the cause, the appearance of a <code>301 Moved Permanently</code> within your own web application is a strong indication that you may need an error management tool to help you automatically detect such errors in the future. The best of these tools can even alert you and your team <em>immediately</em> when an error occurs.
